# Authoring Agent Instructions

> **Claude Code**: All MCP tools are available immediately after the server is connected via `.mcp.json`. No activation steps are needed.

---

## Workflow Overview

The authoring agent reads source documents (Word, Excel, PDF), analyses their structure, and authors them as checklists, queries, or letters in Caseware Cloud via MCP.

**Single-pass pipeline:** Upload → Read content (use `convert_document` for binary formats, read PDF natively) → Analyse structure → Produce procedure outline → `build_payload` → Submit to Caseware Cloud.

## Core Rules

- Never change original document wording or structure; only convert format
- Go directly from reading content to producing a procedure outline — no intermediate markdown or depth-marked markdown step
- Remove leading bullet/number indicators — Caseware Cloud auto-numbers. Stripping is automatic during `build_payload` for procedures, headings, conclusions, and query questions. Covered patterns: `1.` `1.1.` `a.` `B.` `ii.` `IV.` `1)` `a)` `ii)` `(1)` `(a)` `(iii)` `•` `–` `—` `*` `-` (with optional leading whitespace). Bare single letters without dot (e.g. `A ` or `I `) are NOT stripped — too likely to be articles/pronouns. Groups and questionSets are excluded (may have legitimate numbered titles). **Always use `build_payload` — never construct checklist payloads manually, or stripping will not occur.**
- All `-save` MCP operations MUST be sequential (never parallel)
- Execute continuously without pausing for user confirmation. **All responses must be delivered inline in the chat reply** — never defer the answer or require the user to re-prompt.
- Every procedure MUST have a response type configured
- **Conclusions are auto-generated by the API** — do NOT include any conclusion item in the outline. The `build_payload` tool sets `includeConclusion: true` automatically on every checklist payload. Adding a conclusion manually causes it to render as a group, not a platform conclusion.
- Include engagement signoff on procedures unless instructed otherwise

## Depth Marker System (Hierarchy Rules)

| Marker | Purpose | Valid Caseware Types |
|--------|---------|---------------------|
| `[D0]` | Top-level container | `group` |
| `[D1]` | First nesting level | `group` (sub-group), `heading`, `procedure` |
| `[D2]` | Second nesting level | `heading`, `procedure` |
| `[D3]` | Third nesting level | `heading`, `procedure` |
| `[D4]` | Fourth nesting level (max) | `procedure` only |
| `[R]` | Response configuration | Indicates procedure requires response type |
| `[C]` | Conclusion (auto-generated) | Do NOT include — API generates via `includeConclusion: true` |
| `[G]` | Guidance text | Populates guidance field, not a procedure |

### Depth-to-Type Mapping

| Depth | Type Options | Selection Criteria |
|-------|--------------|-------------------|
| 0 | `group` | Default for containers (`[D0]`) |
| 1 | `group` | Sub-group pattern (nesting groups within groups) |
| 1 | `heading` | Default for `[D1]` — section titles under a group |
| 1 | `procedure` | If `[R]` marker present and no children |
| 2-3 | `heading` | If has children below it |
| 2-3 | `procedure` | If `[R]` marker present (actionable item) |
| 4 | `procedure` | Always procedure — maximum depth |

### Structural Patterns

**Pattern 1: Simple Flat (2 levels)**
```
[D0] Group Title
  [D1] Procedure text [R]
```

**Pattern 2: Standard (3 levels)**
```
[D0] Group Title
  [D1] Heading Title
    [D2] Procedure text [R]
```

**Pattern 3: Nested Headings (4 levels)**
```
[D0] Group Title
  [D1] Major Heading
    [D2] Sub-Heading
      [D3] Procedure text [R]
```

**Pattern 4: Sub-Groups (5 levels)**
```
[D0] Group Title
  [D1] Sub-Group Title
    [D2] Heading Title
      [D3] Sub-Heading
        [D4] Procedure text [R]
```

### Validation Rules

- Maximum depth is 4 (no `[D5]` or higher)
- Depth only increases by 1 at each level (no `[D0]` -> `[D3]` jumps)
- All `[D0]` items are `group` type
- `[D4]` items are always `procedure` type
- Every branch terminates in a procedure with `[R]` marker
- `group` type only at depth 0-1

**Invalid Patterns (HARD FAIL):** depth jump > 1, `[D5]+`, `group` at depth 2+, procedure without `[R]`

### Marker Application Rules (Conversion)

| Content Type | Marker | Notes |
|--------------|--------|-------|
| Section/group titles | `[D0]` | Top-level containers |
| Numbered items (1., 2.) | `[D1]` or deeper | Based on nesting level |
| Lettered items (a., b.) | `[D2]`, `[D3]`, or `[D4]` | Based on nesting level |
| Sub-lettered (i., ii.) | Next depth level | Never exceed `[D4]` |
| Conclusions/summaries | — | Auto-generated by API, do not include |
| Instructions/guidance | `[G]` | Populates guidance field |
| Actionable items | Add `[R]` | All procedures need response type |

> **Auto-stripping:** All leading numbering and bullet indicators (`1.`, `a.`, `ii.`, `1)`, `a)`, `ii)`, `(1)`, `(a)`, `(iii)`, `•`, `–`, `—`, `*`, `-`) are automatically stripped from procedure/heading/conclusion text and query question text during `build_payload`. Bare single letters without a dot (e.g. `A `, `I `) are NOT stripped. No manual removal needed in the outline.

### Document Type Detection

| Pattern | Type | Routing |
|---------|------|---------|
| `[D0]`-`[D4]`, `[R]`, `[C]` markers | **Checklist** | checklist-save |
| Question sets with response columns | **Query** | query-save |
| To/From/Date/Re format, salutation/closing | **Letter** | statement-save |

## HARD FAIL Conditions

**CRITICAL:** If any of these errors occur, **STOP immediately** and report:

| Error Code | Meaning | Action |
|------------|---------|--------|
| `CONTENT_LOSS_EXCEEDED` | Parsing lost >10% of procedures | HALT, report expected vs parsed count |
| `MISSING_RESPONSE_TYPE` | Procedure has no `rows` array | HALT, report procedure title, fix before retry |
| `INVALID_DEPTH` | Depth exceeds 4 or depth jump > 1 | HALT, report invalid depth sequence |
| `INVALID_TYPE_AT_DEPTH` | Type not allowed at specified depth | HALT (e.g., group at depth 2+) |
| `INVALID_ID_FORMAT` | ID not 22 chars uuid-base64url | HALT, report invalid ID |
| `NUMBER_CONFLICT` | Document number already exists | HALT, call document-manager with includeHidden:true first |
| `CHUNK_ID_NULL` | First chunk returned null ID | HALT, retry or report MCP issue |
| `PAYLOAD_TRUNCATED` | Response contains fewer procedures than submitted | HALT, reduce batch size and retry |
| `MCP_RETRY_EXHAUSTED` | 3 consecutive MCP failures | HALT after retry logic exhausted |
| `TOOL_BUDGET_EXCEEDED` | Exceeded MCP call budget | HALT, report budget status |

## MCP Payload Validation (Pre-Submission Checklist)

Before any MCP call, verify:
- [ ] Every procedure has `rows` array with at least one `column`
- [ ] All `depth` values are 0-4 (maximum depth is 4)
- [ ] Depth sequence is valid (no jumps > 1, e.g., 0 -> 2 is invalid)
- [ ] Type is valid for depth (depth 0: group, depth 1: group/heading/procedure, depth 2-3: heading/procedure, depth 4: procedure only)
- [ ] Document number does not exist (call `document-manager` with `includeHidden: true`)
- [ ] If ID provided, format is 22-char uuid-base64url
- [ ] `documentInfo` present for new documents
- [ ] `purpose` and `purposeSummary` populated
- [ ] Chunking handled by `build_payload` — follow `submission_instructions`
- [ ] Previous chunk ID captured (for chunks 2+)
- [ ] After each `checklist-save`, compare submitted procedure count to response procedure count — HARD FAIL on mismatch (`PAYLOAD_TRUNCATED`)

## MCP Response Validation (Post-Submission)

After each MCP call:
- Verify response contains expected `id` field (not null for updates)
- Check for error indicators in response
- Validate returned procedure count if available
- Log: `"MCP Response: {status}, id={id}, procedures={count}"`

### Guidance Verification (Post-Save)

After any `-save` call that included guidance fields:

1. **Response contains guidance** → Log `"✓ Guidance confirmed in response"`. Do NOT retry.
2. **Response is ambiguous or timed out** → Log `"⚠️ Guidance submission result unclear — assuming success, verify manually"`. Do NOT retry. Do NOT report failure to the user. Continue to the next operation.
3. **Only report failure** if a separate explicit read-back MCP call confirms the guidance field is empty. Log `"❌ Guidance not added: read-back confirmed empty for [procedure]"`.

**Rules:**
- Never treat a timeout or ambiguous response as proof that guidance was not saved
- Never retry a guidance save unless a read-back confirms the field is empty
- If read-back is unavailable (e.g., no read tool, timeout on read), default to assuming success
- Do NOT use phrases like "guidance may not have been added" or "retrying guidance" without read-back confirmation

## Retry & Recovery Strategy

| Error Type | Retry? | Max Retries | Action |
|------------|--------|-------------|--------|
| Network timeout | Yes | 3 | Exponential backoff (1s, 2s, 4s) |
| Rate limit (429) | Yes | 3 | Wait for Retry-After header |
| Server error (5xx) | Yes | 2 | Exponential backoff (2s, 4s) |
| Authentication (401) | No | 0 | **HARD FAIL** immediately |
| Bad request (400) | No | 0 | **HARD FAIL** (payload issue) |
| Conflict (409) | No | 0 | **HARD FAIL** (document exists) |
| Guidance ambiguous/timeout | No | 0 | Assume success, log `⚠️`, continue |

## Autonomous Execution Mandate

**NEVER pause to ask the user for clarification during workflow execution.** Use best judgment and apply FALLBACK rules when uncertain. The ONLY reasons to stop are HARD FAIL conditions.

**Decision Defaults:**

| Ambiguity | Default Decision | Log Message |
|-----------|------------------|-------------|
| Document type unclear | Checklist | "Defaulting to checklist (most common)" |
| Response type unclear | PICKLIST Yes/No/Not Applicable | "Fallback response type applied" |
| Missing group title | "[Section Title Required]" | "Placeholder group inserted" |
| Missing guidance | "Complete this procedure." | "Default guidance applied" |
| Folder not specified | Use first folder in phase | "Default folder selected" |
| Checklist name not specified | Derive from document/folder name (e.g., "Audit Checklist") | "Default checklist name derived from context" |
| Checklist content not specified | Create 3 placeholder procedures: "Confirm scope and objectives", "Perform and document test procedures", "Document conclusions and exceptions" | "Default placeholder procedures applied" |

**Prohibited Phrases:** Do not use "ask the user", "confirm with user", "wait for input", "check with user", "What should it be called?", "What should I name it?", "What kind of content?", "What do you want inside it?". Replace all such logic with autonomous decision-making.

## Evaluator-Optimizer Loop

Run evaluation before each MCP submission:

**Evaluation Criteria:**
1. Every [R] marker has a response type configured
2. Response types match procedure intent (Priority 1-4 rules)
3. Depth hierarchy is valid (no jumps >1, max depth 4)
4. Types are valid for each depth level
5. Guidance [G] is populated where marked
6. Fallback usage is minimized (flag if >20% procedures use fallback)

**Loop:** If evaluation fails -> apply fixes -> re-evaluate -> max 3 iterations -> if still failing, log warnings and proceed (soft fail).

**Quality Targets:** Response type accuracy >95% | Depth hierarchy validity 100% | Content preservation 100%

## Response Type Detection

**CRITICAL: Apply rules strictly in priority order (1 → 2 → 3 → 4). Stop at the FIRST match.** Yes/No/Not Applicable is the absolute last resort — most procedures should resolve at Priority 1, 2, or 3. If >20% of procedures use fallback, re-evaluate your detection.

### Priority 1: Content Contains Explicit Response Options

Extract inline choices (e.g., "public or private?" → ["Public", "Private", "Not Applicable"]). Always append "Not Applicable" unless present. **If choices are found, use them — do NOT fall through to lower priorities.**

### Priority 2: Keyword Pattern Matching (First Word/Phrase)

Check the **first word or phrase** of the procedure text. Match the FIRST rule that applies — do not skip to Yes/No/N/A. Text/number/date rows are listed BEFORE choice rows intentionally — try them first.

| Pattern (starts with) | Response Type | Configuration |
|-----------------------|---------------|---------------|
| "Describe", "Explain", "Document", "List", "Provide details", "Summarize", "Outline", "Identify", "Obtain", "Prepare", "Draft", "Detail", "Set out", "Indicate" | `text` | `"type": "text"`, `"includeNote": false` |
| "Enter", "Specify", "State", "Record", "Note the", "Name", "Report", "Disclose", "Write", "Complete" | `text` | `"type": "text"`, `"includeNote": false` |
| "Calculate", "Determine the amount", "Compute", "What is the total", "How many", "Quantify", "Estimate the", "What percentage", "What proportion" | `number` | `"type": "number"` |
| "When did", "What date", "On what date", "Date of", "When was", "When is", "What is the date" | `date` | `"type": "date"` |
| "Verify", "Confirm", "Has", "Did", "Does", "Is", "Are", "Was", "Were", "Have", "Can", "Should", "Would", "Will" | `choice` | Yes/No/Not Applicable |
| "Review", "Assess", "Evaluate", "Check", "Inspect", "Examine" | `choice` | Yes/No/Not Applicable |

### Priority 3: Semantic Analysis (Question Structure)

If no keyword matched at Priority 2, analyse the full procedure text. Try text/number/date BEFORE choice:

| Pattern | Response Type |
|---------|---------------|
| Procedure asks for description, explanation, or narrative response | `text` |
| Procedure asks for quantity, count, or monetary amount | `number` |
| Procedure asks for a specific date or time period | `date` |
| Question contains "or" with 2-3 noun options | `choice` (extract options + Not Applicable) |
| Question ends with "?" and asks for yes/no confirmation | `choice` (Yes/No/Not Applicable) |

### Priority 4: Fallback (TRUE Last Resort — Use Only When Priorities 1-3 All Failed)

- Use PICKLIST: "Yes", "No", "Not Applicable"
- **MUST log:** `"⚠️ Fallback: No pattern matched for '[procedure text]', using Yes/No/Not Applicable"`
- If >20% of procedures use fallback, STOP and re-evaluate your Priority 2/3 analysis

### Self-Check Examples

| Procedure Text | Correct Type | Why |
|----------------|-------------|-----|
| "Describe the client's industry and principal activities" | `text` | P2 — starts with "Describe" |
| "How many employees does the entity have?" | `number` | P2 — starts with "How many" |
| "Was the bank reconciliation completed and reviewed?" | `choice` (Yes/No/N/A) | P2 — starts with "Was" |
| "What date was the audit report signed?" | `date` | P2 — starts with "What date" |
| "Obtain details of all related party transactions" | `text` | P2 — starts with "Obtain" |
| "Identify the key risks affecting the entity" | `text` | P2 — starts with "Identify" |

### Critical Payload Rules

- **Choices use `"text"` not `"label"`** — e.g., `{ "text": "Yes" }` not `{ "label": "Yes" }`
- **Use `"type": "text"` for manual entry** — NOT `"type": "manual"`
- **Set `"includeNote": false` for text responses** to avoid duplicate input fields
- **Omit all ID fields** — server auto-generates IDs
- **Do NOT include numbering** in `text` fields — Caseware auto-numbers
- **Choice delimiter is pipe `|`** — use `choice:Option A|Option B|Not Applicable` in response notation. This allows choices to contain commas (e.g., `choice:Completed - no exceptions|Completed with exceptions|Not Applicable`). Comma-delimited format is accepted for backward compatibility only.

## Orchestrator-Workers Pattern (XL Files 400+ Procedures)

Files with **400+ procedures** use orchestrator-workers:

**Orchestrator:** Analyzes structure -> creates chunking plan -> coordinates workers -> verifies completion.
**Workers:** Receive chunk assignment -> build payload -> submit to MCP -> report back.

## Document Reading Rules

- For .pdf files, read natively using the `Read` tool (Claude reads PDFs directly — both text and visual layout)
- For .md files, read directly using the `Read` tool
- For binary files (.docx, .xlsx, .pptx), use the `convert_document` tool to extract text content — Claude Code cannot read these natively
- After reading content, proceed directly to producing a procedure outline for `build_payload`
- Do NOT produce intermediate markdown or depth-marked markdown
- Perform document type detection per `references/document-type-detection.md` after reading

## Unstructured Text

- Review unstructured text and determine how to incorporate it
- Create additional headings or procedures if necessary
- If it cannot be incorporated, mark as [G] (Guidance)

## Checklist Authoring (Summary)

- Never change markdown wording; only author using MCP tools
- Never include procedure summaries — omit the `summary` field or pass empty string
- All picklist/multi-picklist responses MUST include "Not Applicable" as the final choice
- `notePlaceholder` MUST always be "Response and Comments"
- **Checklist Settings:** Set `default_response` at the outline level (e.g. `"default_response": "choice:Yes|No|Not Applicable"`) to apply a default response type to all procedures that don't specify one. Procedures with an explicit `response` field are not overridden.
- Chunking and submission handled by `build_payload` — follow `submission_instructions`
- **Left-align:** Procedure and conclusion text is automatically wrapped in `<p style="text-align: left;">` by `build_payload`. Groups and headings are not wrapped.
- **Dynamic elements in procedures:** Formulas (`engprop()`, `collaborate()`, `wording()`) and known placeholders (`[Date]`, `[Select Staff]`, `[Enter...]`) are auto-detected in procedure text. Generic `[bracketed text]` is NOT converted in checklists (only in letters) to avoid false positives like `[ISA 315]`.

## Query Authoring (Summary)

- Depth 0: Question sets (section headers) — `type: "questionSet"`
- Depth 1: Questions with two columns (text input + file upload) — `type: "question"`
- Use related document number with "Q" suffix
- First retrieve an existing query of the same type as format reference

## Letter Authoring (Summary)

- Use `statement-save` with `type: "letter"`
- Flat structure (all sections at depth 0, type "content")
- **CRITICAL: Two-step creation** — Step 1 creates document map, Step 2 adds sections using captured doc ID and area ID
- Detect and convert all `[bracketed text]` to appropriate dynamic HTML elements (formulas, date/input/staff placeholders)

## Financial Statements (Not Implemented)

If detected: notify user, suggest manual authoring, continue.
