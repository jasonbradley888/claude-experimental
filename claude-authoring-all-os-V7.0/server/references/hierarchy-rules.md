# Hierarchy Rules Reference

Single source of truth for the unified depth marker system used across all authoring workflows.

---

## Depth Marker System

| Marker | Purpose | Valid Caseware Types |
|--------|---------|---------------------|
| `[D0]` | Top-level container | `group` |
| `[D1]` | First nesting level | `group` (sub-group), `heading`, `procedure` |
| `[D2]` | Second nesting level | `heading`, `procedure` |
| `[D3]` | Third nesting level | `heading`, `procedure` |
| `[D4]` | Fourth nesting level (max) | `procedure` only |
| `[R]` | Response configuration | Procedure requires a response — see `load_reference('checklist-authoring')` for type rules |
| `[C]` | Conclusion (auto-generated) | Do NOT include — API generates via `includeConclusion: true` |
| `[G]` | Guidance text | Populates guidance field, not a procedure |

---

## Depth-to-Type Mapping

| Depth | Type Options | Selection Criteria |
|-------|--------------|-------------------|
| 0 | `group` | Default for containers (`[D0]`) |
| 1 | `group` | Sub-group pattern (nesting groups within groups) |
| 1 | `heading` | Default for `[D1]` — section titles under a group |
| 1 | `procedure` | If `[R]` marker present and no children |
| 2-3 | `heading` | If has children below it |
| 2-3 | `procedure` | If `[R]` marker present (actionable item) |
| 4 | `procedure` | Always procedure — maximum depth |

---

## Structural Patterns

Select based on document complexity:

**Pattern 1: Simple Flat (2 levels)**
```
[D0] Group Title
└── [D1] Procedure text [R]
```

**Pattern 2: Standard (3 levels)**
```
[D0] Group Title
└── [D1] Heading Title
    └── [D2] Procedure text [R]
```

**Pattern 3: Nested Headings (4 levels)**
```
[D0] Group Title
└── [D1] Major Heading
    └── [D2] Sub-Heading
        └── [D3] Procedure text [R]
```

**Pattern 4: Sub-Groups (5 levels)**
```
[D0] Group Title
└── [D1] Sub-Group Title
    └── [D2] Heading Title
        └── [D3] Sub-Heading
            └── [D4] Procedure text [R]
```

---

## Validation Rules

Before authoring, verify:
- Maximum depth is 4 (no `[D5]` or higher)
- Depth only increases by 1 at each level (no `[D0]` → `[D3]` jumps)
- All `[D0]` items are `group` type
- `[D4]` items are always `procedure` type
- Every branch terminates in a procedure with `[R]` marker
- `group` type only at depth 0-1

### Invalid Patterns (HARD FAIL)
- Depth jump > 1 (e.g., `[D0]` → `[D2]` without `[D1]`)
- `[D5]` or higher
- `group` type at depth 2+
- Multiple `[C]` markers
- Procedure without `[R]` marker

---

## Marker Application Rules (Structure Analysis)

When the LLM analyses source document content:

| Content Type | Marker | Notes |
|--------------|--------|-------|
| Section/group titles | `[D0]` | Top-level containers |
| Numbered items (1., 2.) | `[D1]` or deeper | Based on nesting level |
| Lettered items (a., b.) | `[D2]`, `[D3]`, or `[D4]` | Based on nesting level |
| Sub-lettered (i., ii.) | Next depth level | Never exceed `[D4]` |
| Conclusions/summaries | — | Auto-generated by API, do not include |
| Instructions/guidance | `[G]` | Populates guidance field |
| Actionable items | Add `[R]` | All procedures need response type |

## Depth Tree Preview Format

Generate before authoring:
```
📋 Depth Tree Preview:
[D0] Group: Risk Assessment Procedures
  [D1] Heading: Understanding the Entity
    [D2] Procedure: Obtain understanding of internal control [R]
    [D2] Procedure: Assess control environment [R]
  [D1] Heading: Fraud Risk Factors
    [D2] Procedure: Inquire of management about fraud [R]
```
> **Note:** Conclusions are auto-generated by the API via `includeConclusion: true` — do NOT include `[C]` items in the outline.

When authoring, remove bullet indicators (a., b., 1., 2., i., ii., etc.) as Caseware Cloud auto-numbers.

---

## Response Types for `[R]` Markers

The `[R]` marker means "this procedure needs a response type configured." It does NOT mean the response is always a picklist. Determine the correct response type using Priority 1-4 rules.

> **Response Types:** See `load_reference('checklist-authoring')` for full Priority 1-4 response type detection rules, keyword tables, and payload configuration examples.

## Document Type Detection

| Pattern | Type | Routing |
|---------|------|---------|
| `[D0]`-`[D4]`, `[R]`, `[C]` markers | **Checklist** | → checklist-save |
| Question sets with response columns | **Query** | → query-save |
| To/From/Date/Re format, salutation/closing | **Letter** | → statement-save |
| Financial statement tables/notes | **Financial Statement** | → Not yet implemented |
