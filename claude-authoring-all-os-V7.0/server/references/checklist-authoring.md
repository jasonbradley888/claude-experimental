# Checklist Authoring Reference

Single source of truth for checklist payload structure, response type detection, and chunking rules.

> **Outline Format:** When authoring checklists, use the `build_payload` tool with a procedure outline instead of building JSON payloads manually. The tool handles all boilerplate (rows/columns/choices, includeNote, notePlaceholder, inlineChoices, chunking). See `load_reference('outline-format')` for the outline schema. The response type detection rules below (Priority 1-4) are still used by the LLM to determine the `response` field in the outline.

---

## General Rules

- Never change the source document wording; only author using MCP tools
- Never include procedure summaries
- Conclusions are auto-generated by the API via `includeConclusion: true` (set by `build_payload`). Do NOT add explicit conclusion items in the procedures array — they render as groups, not conclusions.
- Always set engagement signoff (`includeSignOffs: true`) unless instructed otherwise
- Strip numbering prefixes (1., a., i., 1), a), ii)) — Caseware Cloud auto-numbers based on hierarchy. Bare letters without dot (A, I) are not stripped.
- Populate guidance section with `[G]` content
- Add placeholder titles if groups/headings have no titles
- Do not pause between checklists during authoring
- **Every procedure MUST have a response type** — HARD FAIL if missing

---

## Response Type Detection (Smart Inference)

**CRITICAL: Apply rules strictly in priority order (1 → 2 → 3 → 4). Stop at the FIRST match.** Yes/No/Not Applicable is the absolute last resort — most procedures should resolve at Priority 1, 2, or 3. If >20% of procedures use fallback, re-evaluate your detection.

### Priority 1: Content Contains Explicit Response Options

When a procedure contains inline choices, extract them. **If choices are found, use them — do NOT fall through to lower priorities.**

| Example Procedure | Extracted Choices |
|-------------------|-------------------|
| "Is the entity public or private?" | ["Public", "Private", "Not Applicable"] |
| "Select the accounting framework: GAAP, IFRS, or Other" | ["GAAP", "IFRS", "Other", "Not Applicable"] |
| "Indicate whether the balance is debit or credit" | ["Debit", "Credit", "Not Applicable"] |

**Extraction Rules:**
- Look for "or" connecting 2-3 options within the question
- Look for colon followed by comma-separated options
- Always append "Not Applicable" as the final option unless already present
- Preserve original order of options

### Priority 2: Keyword Pattern Matching (First Word/Phrase)

Check the **first word or phrase** of the procedure text. Match the FIRST rule that applies — do not skip to Yes/No/N/A. Text/number/date rows are listed BEFORE choice rows intentionally — try them first.

| Pattern (starts with) | Response Type | Configuration |
|-----------------------|---------------|---------------|
| "Describe", "Explain", "Document", "List", "Provide details", "Summarize", "Outline", "Identify", "Obtain", "Prepare", "Draft", "Detail", "Set out", "Indicate" | `text` | `"type": "text"`, `"includeNote": false` |
| "Enter", "Specify", "State", "Record", "Note the", "Name", "Report", "Disclose", "Write", "Complete" | `text` | `"type": "text"`, `"includeNote": false` |
| "Calculate", "Determine the amount", "Compute", "What is the total", "How many", "Quantify", "Estimate the", "What percentage", "What proportion" | `number` | `"type": "number"` |
| "When did", "What date", "On what date", "Date of", "When was", "When is", "What is the date" | `date` | `"type": "date"` |
| "Verify", "Confirm", "Has", "Did", "Does", "Is", "Are", "Was", "Were", "Have", "Can", "Should", "Would", "Will" | `choice` | Yes/No/N/A |
| "Review", "Assess", "Evaluate", "Check", "Inspect", "Examine" | `choice` | Yes/No/N/A |

### Priority 3: Semantic Analysis (Question Structure)

If no keyword matched at Priority 2, analyse the full procedure text. Try text/number/date BEFORE choice:

| Pattern | Response Type | Configuration |
|---------|---------------|---------------|
| Procedure asks for description, explanation, narrative | `text` | `"type": "text"`, `"includeNote": false` |
| Procedure asks for quantity, count, monetary amount | `number` | `"type": "number"` |
| Procedure asks for specific date or time period | `date` | `"type": "date"` |
| Question contains "or" with 2-3 noun options | `choice` | Extract options + N/A |
| Question asks "which of the following" | `choice` | Extract listed options + N/A |
| Question ends with "?" and asks for yes/no confirmation | `choice` | Yes/No/N/A |

### Priority 4: Fallback (TRUE Last Resort — Use Only When Priorities 1-3 All Failed)

- Use PICKLIST with options: "Yes", "No", "Not Applicable" (in that order)
- **MUST log:** `"⚠️ Fallback: No pattern matched for '[procedure text]', using Yes/No/Not Applicable"`
- If >20% of procedures use fallback, STOP and re-evaluate your Priority 2/3 analysis

### Self-Check Examples

| Procedure Text | Correct Type | Why |
|----------------|-------------|-----|
| "Describe the client's industry and principal activities" | `text` | P2 — starts with "Describe" |
| "How many employees does the entity have?" | `number` | P2 — starts with "How many" |
| "Was the bank reconciliation completed and reviewed?" | `choice` (Yes/No/N/A) | P2 — starts with "Was" |
| "What date was the audit report signed?" | `date` | P2 — starts with "What date" |
| "Obtain details of all related party transactions" | `text` | P2 — starts with "Obtain" |
| "Identify the key risks affecting the entity" | `text` | P2 — starts with "Identify" |

### Unrecognised Response Types

- Any unrecognised response type string (e.g., "Entity information") defaults to `text` with the original value as placeholder. Log a warning: `"⚠️ Unrecognised response type '[value]', defaulting to text"`
- Handle `[Non optimal]` variants in option strings: strip the `[Non optimal]` suffix and parse remaining options normally (e.g., `Yes | No [Non optimal]` → choices `Yes`, `No`)

## Payload Structure

> **Reference only.** Do NOT manually construct these JSON payloads. Use `build_payload(outline)` — it generates the correct structure, validates it, and returns `submission_instructions` telling you exactly how many API calls to make. Follow those instructions precisely.

### Top-Level Payload

```json
{
  "id": null,
  "documentInfo": {
    "folder": "uuid-base64url",
    "name": "Checklist Name",
    "number": "DOC-XXX"
  },
  "guidance": "Guidance text from [G] markers",
  "includeConclusion": true,
  "conclusionTitle": "Conclusion",
  "procedures": [...],
  "purpose": "Description of why checklist is being created",
  "purposeSummary": "Short title for suggestion set"
}
```

### Procedure: CHOICE Response (Yes/No/N/A)

```json
{
  "depth": 2,
  "type": "procedure",
  "text": "Has management approved the financial statements?",
  "hidden": false,
  "includeNote": true,
  "notePlaceholder": "Response and comments",
  "includeSignOffs": true,
  "rows": [{
    "columns": [{
      "type": "choice",
      "placeholder": "",
      "includeOtherChoice": false,
      "inlineChoices": true,
      "choices": [
        { "text": "Yes" },
        { "text": "No" },
        { "text": "Not Applicable" }
      ]
    }]
  }]
}
```

### Procedure: TEXT Response (Free-Form Entry)

```json
{
  "depth": 1,
  "type": "procedure",
  "text": "Describe the client's business and primary revenue streams",
  "hidden": false,
  "includeNote": false,
  "includeSignOffs": true,
  "rows": [{
    "columns": [{
      "type": "text",
      "placeholder": "Enter description here"
    }]
  }]
}
```

### Procedure: PICKLIST Response (Multiple Options)

```json
{
  "depth": 2,
  "type": "procedure",
  "text": "What is the assessed level of inherent risk?",
  "hidden": false,
  "includeNote": true,
  "notePlaceholder": "Response and comments",
  "includeSignOffs": true,
  "rows": [{
    "columns": [{
      "type": "choice",
      "placeholder": "",
      "includeOtherChoice": false,
      "inlineChoices": true,
      "choices": [
        { "text": "Low" },
        { "text": "Moderate" },
        { "text": "High" },
        { "text": "Not Applicable" }
      ]
    }]
  }]
}
```

### GROUP (depth 0, no response)

```json
{
  "depth": 0,
  "type": "group",
  "text": "Revenue Recognition",
  "hidden": false,
  "includeNote": true,
  "includeSignOffs": true,
  "notePlaceholder": "Response and comments"
}
```

### HEADING (depth 1, no response)

```json
{
  "depth": 1,
  "type": "heading",
  "text": "Understanding the Process",
  "hidden": false,
  "includeNote": true,
  "includeSignOffs": true,
  "notePlaceholder": "Response and comments"
}
```

### CONCLUSION

> **Do NOT add conclusion items to the procedures array.** Set `"includeConclusion": true` on the top-level payload — the API generates the real conclusion widget automatically. Explicit conclusion items render as groups, not actual conclusions. The `build_payload` tool strips any conclusion items from the outline automatically.

## Critical Payload Requirements

> **These rules are enforced by `build_payload` automatically.** They are listed here for reference only — do NOT apply them manually. Use `build_payload(outline)` and follow its `submission_instructions`.

- Choices use `"text"` not `"label"` — e.g. `{ "text": "Yes" }` not `{ "label": "Yes" }`
- Use `"type": "text"` for text entry — NOT `"type": "manual"` (see warning below)
- Do NOT add conclusion items — the API generates conclusions via `includeConclusion: true`

### ⚠️ CRITICAL: Text Entry Response Type

- Use `"type": "text"` for manual/free-form text entry responses
- Do **NOT** use `"type": "manual"` — this type does not create response columns correctly
- When using text responses, set `"includeNote": false` to avoid duplicate input fields
- The `placeholder` field provides guidance text inside the text input field

## Chunking

`build_payload` handles chunking automatically when >50 procedures — you do not need to chunk manually.
The result includes `submission_instructions` — follow them exactly for the correct number of API calls.

---

## Post-Authoring Verification

After each `checklist-save` call:
1. Verify response contains expected `id` field (not null for updates)
2. Compare response procedure count against `expected_per_payload` from metadata — HARD FAIL with `PAYLOAD_TRUNCATED` if they differ
3. Log: `"MCP Response: {status}, id={id}, procedures={count}"`

> **Note:** `build_payload` handles all validation (depth sequences, types at depth, response completeness, duplicate detection, empty branches, content loss). No separate evaluator-optimizer loop is needed.

## Checklist Settings (default_response)

Set `default_response` at the outline level to apply a default response type to all procedures without an explicit `response` field. This simplifies outlines where most procedures share the same type.

```json
{
  "document_type": "checklist",
  "default_response": "choice:Yes|No|Not Applicable",
  "items": [
    {"depth": 0, "type": "group", "text": "Section"},
    {"depth": 1, "type": "procedure", "text": "Check it"},
    {"depth": 1, "type": "procedure", "text": "Describe it", "response": "text"}
  ]
}
```

- The first procedure above inherits `choice:Yes|No|Not Applicable`
- The second procedure keeps its explicit `text` response
- Works in both create and update modes (update applies to new items only)

## Dynamic Elements in Procedures

`build_payload` auto-detects formulas and known placeholders in procedure text and converts them to Caseware HTML:

| Pattern in text | Converted to |
|----------------|-------------|
| `entity name`, `company name`, `client name` | `engprop("name")` formula |
| `year end`, `period end`, `balance sheet date` | `engprop("yearend", 0, "longDate")` formula |
| `firm name`, `auditor firm` | `collaborate("firmName")` formula |
| `[Date]`, `[Letter Date]`, `[Select Date]` | Date placeholder |
| `[Enter...]`, `[Insert...]`, `[Describe...]` | Input-area placeholder |
| `[Select Staff]`, `signature`, `signed by` | Staff selector |
| `wording("@glossaryId")` (explicit formula) | Formula HTML span |

**Important:** Generic `[bracketed text]` like `[ISA 315]` or `[Section 3.1]` is NOT converted in checklists — only known patterns are matched. This prevents false positives. Letters still convert all brackets.

---

## Text Alignment

Procedure and conclusion text is automatically wrapped in `<p style="text-align: left;">` by `build_payload` for consistent left-aligned rendering. Group and heading text is not wrapped (they are structural titles). Already-HTML text (starting with `<`) is not double-wrapped.
